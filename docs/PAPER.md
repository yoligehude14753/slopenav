## Abstract

Self-refining coding agents — Claude Code, OpenAI Codex CLI, Cursor Composer, Aider, AutoGen / CrewAI / Letta loops — face a banal but expensive question on every run: when to stop. Today they stop on a hard-coded iteration budget (e.g. Self-Refine's default 4 [Madaan et al. 2023]) or a fixed score threshold; both waste tokens on tasks that converge fast and cut off tasks that need more time. **SlopeNav** is a small, pure-Python decision algorithm that consumes a sequence of quality scores plus optional per-criterion binary verdicts and emits one of `continue`, `pivot`, or `deliver`. It tracks **two slopes in parallel** (windowed linear regression for global trend, EMA difference for local momentum), adds a **verdict-level progress signal** (which sub-criteria are improving, which are persistently failing), and routes the inputs through a **deterministic 9-rule decision tree**.

We evaluate on three regimes. On 500 synthetic agent traces with known oracle stops, SlopeNav reaches **efficiency = 0.234** (success rate / average iterations used) vs. **0.118** for fixed-5, **2.0×** the efficiency of the most common Self-Refine configuration. On 100 real API trajectories generated by `gpt-4o-mini` Self-Refine on FLASK instructions and scored by QAG-Gate, SlopeNav reaches **efficiency = 0.541** vs. **0.176** for fixed-5 (3.07× ratio, success rate 0.93 vs. 0.88, mean 1.72 iterations vs. 5.0). We also report a negative finding honestly: in this real-trajectory regime the quality scores barely move across iterations (mean ≈ 0.85 at every step), and a simple Δ-threshold baseline reaches **efficiency = 0.664** — *higher* than SlopeNav. SlopeNav's slope-and-verdict design pays off when trajectories are informative (synthetic), and ties or loses to thresholding when they are flat (this particular FLASK × `gpt-4o-mini` regime). A separate **verdict-targeted reprompt study** (E4, n = 32 pivot traces) finds **Δscore = +0.034** vs. a generic reprompt baseline, with Wilcoxon one-sided *p* = 0.065 — not significant — and the two largest gains are RedLine-interaction artefacts rather than evidence that targeted reprompts beat generic ones.

The package has **zero required dependencies**, ships under Apache 2.0 as `pip install slopenav`, and is evaluator-agnostic — it works with QAG-Gate, G-Eval, RAGAS, or any custom score sequence.

**Keywords**: iterative agents, stopping criterion, coding agents, convergence detection, agent observability

## 1. Introduction

When an agent like Claude Code, Codex CLI, or Aider is asked to write or refactor code, it does not produce the final artifact in a single shot. It plans, edits, runs tests, reads stack traces, edits again, and only at some point decides that it is done. In *Self-Refine* [Madaan et al. 2023] and *Reflexion* [Shinn et al. 2023] the loop is explicit; in newer commercial wrappers it is implicit but still iterative. The cost question is identical across all of them: each extra iteration consumes another full LLM round-trip, and at scale (parallel sessions, scheduled re-runs, multi-tenant CI) the difference between stopping at iteration 2 and stopping at iteration 5 is the difference between a profitable feature and an expensive one.

Existing stopping criteria are crude:

1. **Fixed iteration budgets** (Self-Refine ships with 4; Reflexion typically 3–5): trivial to implement, wasteful in both directions — it over-iterates on easy tasks and cuts off on hard ones.
2. **Single scalar threshold** ("stop when score ≥ 0.80"): sensitive to scorer calibration and silently fails when the scorer's distribution shifts.
3. **Confidence-based early exit** [Shridhar et al. 2023]: works in retrieval Q&A where confidence is well-calibrated, harder to extend to open-ended generation quality.
4. **Human-in-the-loop**: the gold standard, but defeats the purpose of unattended long-running agents.

SlopeNav targets the gap between fixed budgets and human review. The core observation is mundane: *the rate of improvement* in a quality score is usually a stronger stopping signal than the absolute level of that score. An agent at score 0.78 that is still climbing should keep going; an agent that has been stuck at 0.84 for three iterations is almost certainly done.

**Contributions.**

1. **Dual-slope tracking** (§3.1): a sliding-window OLS slope for the long-run trend plus an EMA-difference slope for local momentum, with an adaptive "high slope" threshold that scales with the number of active binary criteria (so a single criterion flip has the right effective weight).
2. **Verdict-level progress analysis** (§3.2): on top of the scalar, SlopeNav tracks per-criterion verdict transitions, persistent failures (≥ 3 consecutive `fail`s on the same criterion), and inter-iteration stability. Persistent failures are the cleanest signal for routing to `pivot` instead of either `continue` or `deliver`.
3. **Stagnation diagnoser** (§3.3): when both slopes are non-positive, decide whether the agent has plateaued at its capability limit (deliver with a flag) or whether the *evaluator* is missing improvements (deliver and signal for human re-rating).
4. **Deterministic 9-rule decision tree** (§3.4): a pure function from `(scores, verdicts, context)` to `Decision`. Fully unit-testable, fully debuggable, and small enough that an operator can read the full rule set and predict the behaviour.
5. **Three benchmark regimes** (§5): synthetic traces with known oracle stops (where slope reasoning helps); real `gpt-4o-mini` Self-Refine trajectories on FLASK (where slope reasoning saves iterations but a simpler baseline does even better); a verdict-targeted reprompt study (a negative result we report rather than hide).

## 2. Background

### 2.1 Iterative Coding Agents in Production

Commercial coding agents (Claude Code, Codex CLI, Cursor Composer, Aider, Devin) and the open-source loops they popularise (AutoGPT, AutoGen, CrewAI, Letta) all follow a generate → evaluate → refine cycle. The seminal academic instantiations are Self-Refine [Madaan et al. 2023], which uses a fixed iteration budget (typically 4) and reports diminishing returns after iteration 3–5 on its benchmarks, and Reflexion [Shinn et al. 2023], which uses verbal success / failure signals from an environment but does not analyse the trajectory of a quality score.

### 2.2 Adaptive Stopping in Classical ML

Adaptive stopping has a long history outside LLMs. *Early stopping* in neural network training monitors validation-loss slope [Prechelt 1998]. *Bandit algorithms* balance exploration and exploitation [Lattimore & Szepesvári 2020]. *Sequential hypothesis testing* gives theoretical guarantees for online decisions [Wald 1947]. SlopeNav transports the slope-monitoring intuition from training curves to agent quality curves; the unit of observation is one iteration of the agent rather than one epoch, and the "validation loss" is whatever evaluator the user plugs in.

### 2.3 Convergence Detection for LLM Loops

The closest LLM-specific work is Shridhar et al. [2023], who propose confidence-based early stopping for retrieval Q&A: stop when answer confidence exceeds a threshold. This is effectively a single-scalar threshold rule, similar to our "Δ-threshold" baseline in §5.3. Self-Refine [Madaan et al. 2023] uses fixed iterations; Reflexion [Shinn et al. 2023] uses external success signals but not trajectory analysis.

We are not aware of an open-source stopping criterion that combines (i) two complementary slope signals, (ii) per-criterion verdict tracking, and (iii) capability / evaluation-blind-spot diagnosis under a single deterministic decision tree.

## 3. SlopeNav Framework

### 3.1 Dual-Slope Computation

Let $S = [(t_0, s_0), (t_1, s_1), ..., (t_k, s_k)]$ be the score history, where $s_i \in [0, 1]$ is the quality score at iteration $t_i$. We compute over a sliding window of size $w$ (default 5):

**Linear slope** via ordinary least squares:

$$\hat{\beta}_{lin} = \frac{\sum_{i} (t_i - \bar{t})(s_i - \bar{s})}{\sum_{i} (t_i - \bar{t})^2}$$

**EMA slope** as the difference between consecutive EMA values:

$$\text{EMA}_{k} = \alpha \cdot s_k + (1 - \alpha) \cdot \text{EMA}_{k-1}, \quad \alpha = 0.4$$

$$\hat{\beta}_{ema} = \text{EMA}_{k} - \text{EMA}_{k-1}$$

The dual-slope design provides complementary signals: $\hat{\beta}_{lin}$ captures the global trend over the window (robust to individual outliers), while $\hat{\beta}_{ema}$ captures immediate momentum (responsive to recent changes).

**Adaptive threshold**: The "high slope" threshold $\tau_{high}$ is adapted to the number of active evaluation questions. With $n$ binary questions, a single question flip contributes $1/n$ to the score. We set:

$$\tau_{high} = \max(0.01, \min(0.05, \frac{0.6}{n}))$$

This prevents premature "continue" decisions when many questions are active (each flip has small absolute impact).

### 3.2 Verdict-Level Progress Analysis

Beyond aggregate scores, SlopeNav tracks per-verdict changes. At each iteration, the evaluator produces a list of binary verdicts $V_k = \{v_1, v_2, ..., v_n\}$. We compute:

**Net progress**:
$$\Delta_k = |\{v \in V_k : v = \text{pass}, v \in V_{k-1} : v = \text{fail}\}| - |\{v \in V_k : v = \text{fail}, v \in V_{k-1} : v = \text{pass}\}|$$

**Persistent failures**: A verdict $v_j$ is a *persistent failure* if it has been `fail` for $\geq 3$ consecutive iterations. Persistent failures indicate that the agent is systematically unable to satisfy a specific requirement—the prime signal for a `pivot` decision.

**Stability score**: Fraction of verdicts that did not change between iterations:
$$\text{stability} = 1 - \frac{|\{v : v_k \neq v_{k-1}\}|}{n}$$

High stability (≥ 0.85) combined with a score above the minimum threshold triggers a `deliver` decision.

### 3.3 Stagnation Diagnosis

When all slope signals are non-positive and persistent failures are present, SlopeNav invokes a stagnation diagnoser to distinguish:

**Capability limit**: The agent has genuinely plateaued. Indicators: score in [min\_threshold × 0.85, min\_threshold], score variance over last 5 iterations < 0.02, no recent verdict improvements. Response: `deliver` (accept best achievable output).

**Evaluation blind spot**: The evaluator may be too strict or missing improvements. Indicators: score below 0.60 despite evidence of tool success (tool\_results non-empty), high diversity in recent verdicts (stability < 0.50). Response: `deliver` with flag (signal to caller to request human review or re-evaluation).

### 3.4 Decision Tree (9 Rules)

Rules are evaluated in priority order. First matching rule fires.

| Rule | Condition | Action | Reason |
|------|-----------|--------|--------|
| 0 | n_evals < min_required | continue (unless excellent) | need_slope_data |
| 1 | score ≥ 0.88 AND (stability ≥ 0.7 OR n_evals ≥ 3) | **deliver** | excellent_score |
| 2 | verdict regression AND score < threshold | continue | verdict_regression |
| 3 | score ≥ 0.85 AND lin\_slope < 0.05 AND ema\_slope < 0.03 | **deliver** | good_enough |
| 4 | lin\_slope > τ\_high OR ema\_slope > 0.03 | continue | high_slope_improving |
| 5 | score ≥ threshold AND slope ≤ τ\_high | **deliver** | above_threshold_flat |
| 6 | persistent\_failures ≥ 3 AND n\_evals ≥ 4 | pivot/deliver | persistent_stagnation |
| 7 | n\_evals ≥ 10 AND best\_score ≥ threshold × 0.90 | **deliver** | patience_exhausted |
| 8 | lin\_slope ≤ 0 AND ema\_slope ≤ 0 | pivot/deliver | stagnant_or_declining |
| 9 | (else) | continue | weak_positive_slope |

The decision tree is a *pure function*—given the same inputs, it always produces the same output. This makes it fully unit-testable and interpretable. Rules 0–5 cover 90%+ of real-world cases; Rules 6–9 handle edge cases.

## 4. SlopeNav-Bench

### 4.1 Synthetic Traces

We generate 2,000 synthetic agent traces with known optimal stopping points, covering 5 convergence patterns:

| Pattern | Description | Oracle Stop | Frequency |
|---------|-------------|-------------|-----------|
| Monotonic rise | Score increases steadily | When score ≥ 0.80 | 30% |
| Plateau after rise | Fast rise then plateau | Start of plateau | 25% |
| Stagnation | Mediocre quality, no progress | After n_max | 20% |
| Fast-good | High quality from start | Iteration 1–2 | 15% |
| Decline | Agent regressing | Before regression | 10% |

Each trace has: max_iter ∈ [5, 10], quality scores ∈ [0, 1], ±Gaussian noise (σ=0.04).

### 4.2 Real API Trajectories (E3)

For the external evaluation in §5.3 we generate 100 trajectories on the fly: sample 100 FLASK instructions, run up to 5 Self-Refine iterations with `gpt-4o-mini` as the generator and reviser, and score every iteration with QAG-Gate using a `delivering` context (forcing deep evaluation). No human annotator labels an "optimal stop" for these traces; the oracle in §5.3 is operationally defined as the first iteration that crosses the success threshold (`score ≥ 0.70`), so all reported `Prec@Oracle` numbers should be read as "how often the strategy stops at-or-before the first crossing", not "how often it stops where a human would have".

### 4.3 Evaluation Protocol

**Primary metric**: Efficiency = success\_rate / avg\_iterations\_used

Where success = final score ≥ success\_threshold (0.70) at the stopping iteration.

**Secondary metrics**:
- Iter savings: (fixed-5 avg\_iter − SlopeNav avg\_iter) / fixed-5 avg\_iter
- Quality loss: fixed-5 success\_rate − SlopeNav success\_rate (should be ≤ 0.05)
- Precision@oracle: fraction of traces where SlopeNav stops within ±1 iteration of oracle

## 5. Experiments

### 5.1 Results on Synthetic Traces

**P0 (200 traces, 2 strategies)**:

| Strategy | Avg Iterations | Success Rate | Efficiency |
|----------|----------------|--------------|------------|
| SlopeNav | 3.37 | 0.795 | 0.236 |
| Fixed-3 | 3.00 | 0.300 | 0.100 |

**P1 (500 traces, 4 strategies)**:

| Strategy | Avg Iterations | Success Rate | Efficiency |
|----------|----------------|--------------|------------|
| **SlopeNav** | **3.38** | **0.790** | **0.234** |
| Fixed-3 | 3.00 | 0.320 | 0.107 |
| Fixed-5 | 5.00 | 0.588 | 0.118 |
| Threshold-0.75 | 3.74 | 0.784 | 0.210 |
| Oracle | 5.99 | 0.588 | 0.098 |

Key observations:
- SlopeNav achieves **2.0× efficiency** vs Fixed-5 (0.234 vs 0.118)
- SlopeNav achieves **+47pp success rate** vs Fixed-3 (0.790 vs 0.320) with only +0.38 more iterations
- Threshold-0.75 is competitive but depends on a manually-chosen threshold; SlopeNav requires no calibration beyond min_threshold
- Oracle (perfect hindsight) performs *worse* than SlopeNav in efficiency because oracle's criterion (reach score ≥ 0.80) causes it to run to max on stagnating traces, while SlopeNav's slope-based detection exits earlier

### 5.2 Internal simulated traces (P2, supplementary)

Earlier **P2** runs used **QAG-Gate scores on simulated improvement patterns** (200 traces) to stress-test the decision tree. SlopeNav improved success vs. some fixed budgets in that **synthetic** regime (see v1.0 text). These results are **not** the main claim once E3 exists.

### 5.3 Public-task trajectories — Self-Refine + QAG-Gate (E3, n=100)

We sample **100 FLASK instructions**, run up to **5** Self-Refine iterations with **`gpt-4o-mini`**, and score each iteration with **QAG-Gate** (same protocol as our QAG-Gate paper: delivering context for deep evaluation). Success := final score ≥ threshold in the replication script (`benchmarks/2026-05-real-traj/run_e3.py`).

**Aggregate scores are nearly flat** (mean QAG score ~0.85 every iteration)—the base model already produces strong answers—so stop early is the dominant phenomenon.

| Strategy | Success rate | Avg iterations | Efficiency | Prec@Oracle |
|----------|---------------|----------------|------------|-------------|
| **SlopeNav** | **0.930** | **1.72** | **0.541** | **0.825** |
| Fixed-3 | 0.880 | 3.00 | 0.293 | 0.062 |
| Fixed-4 (Self-Refine default) | 0.910 | 4.00 | 0.228 | 0.041 |
| Fixed-5 | 0.880 | 5.00 | 0.176 | 0.010 |
| **Δ-threshold** (tuned early-stop) | **0.950** | **1.43** | **0.664** | **0.907** |

SlopeNav **reduces iterations sharply** vs. fixed-5 (**3.07×** efficiency ratio in this setup) but **does not beat** the simple Δ-threshold baseline on efficiency here—highlighting that **when scores barely move**, thresholding is extremely competitive; SlopeNav’s slope + verdict design matters most when **trajectories are informative** (as in §5.1 synthetic sets).

### 5.4 Verdict-conditioned reprompt (E4, exploratory, n=32 pivot traces)

Among traces where SlopeNav emitted a **pivot** signal, we compared **generic** vs. **verdict-targeted** reprompts followed by a second QAG evaluation (n=32 pairs).

| Metric | Value |
|--------|-------|
| Mean Δscore (targeted − generic) | 0.034 |
| Median Δscore | 0.000 |
| B > A | 15 / 32 (46.9%) |
| Wilcoxon signed-rank *p* (H₁: targeted > generic) | 0.065 |

**Negative overall result**: Wilcoxon one-sided *p* = 0.065 does not reach α = 0.05; median Δ = 0. Two cases had `score_a = 0.0` from RedLine triggering on the generic prompt—qid=509 produced Δ = +0.818 as a mechanism artefact, not evidence of prompt quality. Removing both score_a < 0.1 outliers (n=30): mean Δ = 0.009, *p* = 0.104—even weaker. Verdict-level diagnostics require more than prompt-level delivery to realise their potential.

### 5.5 Ablation: Which Signal Matters More?

Component ablations on **live** traces are **future work**; synthetic ablations (§5.1 codebase) show dual-slope + verdict hooks working in controlled noise settings.

### 5.6 Sensitivity Analysis (synthetic)

We test SlopeNav's robustness to:
- **Score noise level**: σ = 0.02, 0.04 (default), 0.08 → efficiency degrades gracefully
- **Window size**: w = 3, 5 (default), 8 → w=5 optimal for typical 5–10 iteration traces
- **EMA α**: 0.2, 0.4 (default), 0.6 → larger α (more recent-biased) slightly better for declining traces

## 6. Case Studies

The traces below are taken from the E3 corpus (`gpt-4o-mini` Self-Refine on FLASK instructions, scored by QAG-Gate).

### 6.1 Fast convergence

A FLASK item asking for a short factual list-completion produces score 0.91 at iteration 1. With three baseline-question evaluations in the window, SlopeNav fires Rule 1 ("excellent score with sufficient stability") at iteration 1 and delivers. Fixed-3 continues to iteration 3 and ships the same answer; the two extra LLM round-trips cost roughly 2× the per-task tokens for zero quality gain.

### 6.2 Stagnation at the model's capability ceiling

A multi-step reasoning prompt plateaus at score 0.58 from iteration 2 onwards; the verdict `reasoning_completeness` flips to `fail` and stays there. By iteration 5 SlopeNav fires Rule 6 (persistent failure ≥ 3), runs the stagnation diagnoser, and tags the case as `capability_limit` — slopes are flat *and* score is below the success threshold, so refining further is unlikely to help. SlopeNav delivers at iteration 5 with a flag, identical to Fixed-5's output but with a structured `stagnation` reason that downstream UIs can surface to a human.

### 6.3 Late improvement

A code-generation prompt drifts at the start (score 0.31 at iteration 1, 0.42 at iteration 2, 0.65 at iteration 3) and then jumps to 0.82 at iteration 4 (window OLS slope ≈ 0.17, well above τ_high). SlopeNav fires Rule 4 (`high_slope_improving`) and *does not* deliver yet, even though the score has crossed the success threshold; one more iteration brings the trajectory to 0.85 with a near-flat slope, at which point Rule 3 (`good_enough`) fires and SlopeNav delivers at iteration 5. Fixed-3 would have shipped at score 0.65 (below the success threshold and failing the deliverable gate).

## 7. Discussion

### 7.1 Why Dual Slopes?

In early experiments, linear-only stopping triggered prematurely when a single high-scoring iteration created a downward trend after the peak. EMA-only stopping missed gradual upward trends in noisy traces. The combination provides both robustness and responsiveness.

### 7.2 Integration with Any Evaluator

SlopeNav is evaluator-agnostic. It consumes any sequence of scores and optional binary verdicts. It has been integrated with QAG-Gate scores (recommended) but also works with RAGAS scores, G-Eval scores, or custom heuristics.

```python
# Works with any score source
from slopenav import SlopeNav

nav = SlopeNav(min_threshold=0.80)
for iteration, (score, verdicts) in enumerate(agent_loop()):
    decision = nav.step(iteration=iteration, score=score, verdicts=verdicts)
    if decision.action == "deliver":
        break
```

### 7.3 Limitations

- **Flat trajectories (E3)**: When QAG scores barely change, slope signals are weak; simple thresholds may match or beat slope-based rules—report both.
- **No human oracle stops in E3**: We compare to scripted baselines, not crowd-sourced optimal stop points.
- **E4 sample size (n=32)** and **outliers** (e.g., RedLine interactions) limit statistical strength.
- **Warm-up period**: SlopeNav needs ≥2 scored iterations before slopes stabilize; single-shot tasks need thresholds.

## 8. System Description

SlopeNav is implemented in pure Python with zero required dependencies (numpy optional for ≥10% speedup). Core algorithm runs in microseconds per call.

### 8.1 Package Structure

```
slopenav/
├── domain/    # VerdictSnapshot, SlopeResult, Decision, StagnationDiagnosis
├── slope/     # compute_linear_slope, compute_ema_slope
├── verdicts/  # compute_verdict_progress
├── decision/  # decide() — pure function, 9-rule tree
├── diagnose/  # diagnose_stagnation
└── nav.py     # SlopeNav — stateful wrapper
```

### 8.2 Quick Start

```python
from slopenav import SlopeNav

nav = SlopeNav(min_threshold=0.80, max_pivots=2)

# Typical usage in agent loop
for i, (score, verdicts) in enumerate(agent_evaluations):
    decision = nav.step(iteration=i, score=score, verdicts=verdicts)
    print(f"iter={i} score={score:.3f} → {decision.action} ({decision.reason})")
    if decision.action == "deliver":
        return best_output
    elif decision.action == "pivot":
        change_agent_strategy(decision.stagnation)
```

## 9. Conclusion

SlopeNav is a small, deterministic stopping criterion for iterative coding agents — Self-Refine and Reflexion loops, AutoGen / CrewAI / Letta workflows, and the commercial wrappers (Claude Code, Codex CLI, Cursor Composer, Aider) that internally run the same kind of loop. It combines a windowed OLS slope, an EMA momentum slope, a verdict-level progress signal, and a 9-rule decision tree, in roughly 500 lines of pure Python with no required dependencies. On synthetic traces with informative quality curves it delivers ~2× the efficiency of fixed-5 stopping. On 100 real `gpt-4o-mini` Self-Refine trajectories generated on FLASK instructions it cuts iterations from 5.0 to 1.72 while *improving* success rate (0.93 vs. 0.88), but loses on efficiency to a simple Δ-threshold rule because the underlying score curve is nearly flat in this particular agent / task pairing. A targeted verdict-conditioned reprompt experiment (n=32) did not beat a generic reprompt baseline on average (Wilcoxon *p* = 0.065). We release the package under Apache 2.0 (`pip install slopenav`) and intend to follow up on (a) richer pivot policies that go beyond per-criterion reprompting and (b) calibrating the slope thresholds for *steeper* score regimes than the FLASK × `gpt-4o-mini` setup we benchmarked here.
